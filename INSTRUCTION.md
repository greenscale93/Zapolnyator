Вот моя задача:
# Инструкция: заполнение отчёта ДКП (1С-выгрузка → Excel)

## Назначение
Пользователь ежемесячно присылает: (1) Excel-файл отчёта за предыдущий месяц (защищён паролем) и (2) выгрузку из 1С в формате MXL за новый месяц. Задача Claude — заполнить 4 листа-источника данными нового месяца, не тронув старые проверенные периоды, пересчитать отчёт, проверить корректность и выдать итоговые показатели.

**Пример сценария:** пользователь присылает файл с данными по май включительно + выгрузку за июнь → Claude добавляет июнь. В следующий раз: файл с июнем + выгрузка за июль → Claude добавляет июль.

---

## Роль и стиль
- Отвечать формально, кратко, по делу. Структура: резюме → детали → критика/риски.
- Пользователь — руководитель департамента, эксперт. Не разжёвывать базовое.
- **Всё, в чём есть сомнение — спрашивать, а не додумывать.** Лучше уточнить, чем внести ошибку в отчёт для руководства. Особенно: новые подразделения, новые контрагенты-офисы, новые ставки НДС, новые виды взаиморасчетов.
- Показывать критику и риски отдельным блоком.

---

## Входные файлы

### Excel-файл отчёта
- Имя вида `ДКП_10_-_<Месяц>_<Год>.xlsx`. Защищён **паролем на открытие: 987456** (запрашивать, если не сработал; не хранить как жёсткую константу, но по умолчанию пробовать этот).
- Итоговый файл сохранять с тем же паролем и именем нового месяца.

### MXL-выгрузка (основная)
Текстовый формат MOXCEL (1С «Табличный документ»), UTF-8. Плоская таблица, 38 колонок + «Комментарий». Парсится программно (см. раздел «Парсинг MXL»).

Колонки основной выгрузки по порядку:
1 ТипЗаписи, 2 Дата, 3 Номер, 4 НомерК7, 5 ИДКонтрольнойТочкиКСУП, 6 Документ, 7 Подразделение, 8 ПодразделениеКонтрагент, 9 **ПодразделениеКонтрагентДляОтчета**, 10 Контрагент, 11 Проект, 12 Этап, 13 Сценарий, 14 Сотрудник, 15 Квалификация, 16 Направление, 17 ТипОборота, 18 ВидОборота, 19 ВидВзаиморасчета, 20 ВидНачисленияЗП, 21 СуммаБезНДС, 22 СуммаСНДС, 23 СтавкаНДС, 24 Себестоимость, 25 Оклад, 26 Премия, 27 АдминистративныеРасходы, 28 Сумма, 29 ТрудозатратыЧасы, 30 Грейд, 31 ПроцентПоАктам, 32 ПроцентПоДеньгам, 33 СтавкаНалогаНаФОТ, 34 ОплатаАванса, 35 ВГО, 36 ЭтоВстречныйДокумент, 37 Ответственный, 38 ДокументОснование, (последняя) Комментарий.

Типы записей (колонка ТипЗаписи): `Реализация`, `Оплата`, `Взаиморасчет`, `НачислениеЗарплаты`, `ФактическийФОТ`, а также служебные `ИсторияКвалификации`, `ОкладСотрудника` (последние два не используются).

### MXL-список взаиморасчетов (контрольный, опциональный)
Пользователь может дополнительно прислать отдельный список всех взаиморасчетов (15 колонок: Дата, Номер, Подразделение, Направление, ПодразделениеКонтрагент, Сценарий, Контрагент, СуммаБезНДС, Проект, Этап, Комментарий, ДокументОснование, Ответственный, ВГО, ЭтоВстречный, Вид). Использовать для дополнительной сверки взаиморасчетов, если приложен.

## Общие правила отбора данных (применять ко ВСЕМ типам)

1. **Только сценарий `ФАКТ`.** Все строки с другими сценариями (ПРОГНОЗ, ОПЕР и т.п.) отбрасывать. Исключение: тип `ФактическийФОТ` идёт по своему сценарию — его брать целиком (кроме исключений ниже).

2. **Полностью игнорировать подразделение `ГКП 10.6 (Емельянова)`** — во всех типах и во всех ролях (и когда оно в поле «Подразделение», и когда в «ПодразделениеКонтрагент»). Это подразделение не входит в отчёт департамента.

3. **Пустые строки на листах удалять** — данные должны идти сплошным блоком без разрывов.

4. **Новый месяц записывать в конце листа**, после последней существующей строки. Старые периоды не перезаписывать (см. раздел «Защита старых периодов»).

## Правила по каждому листу

### Лист «Реализация» (тип записи `Реализация`)
Заполнение колонок листа:
- Подразделение ← Подразделение
- Сценарий ← «ФАКТ»
- Период ← текст «`<Месяц> <Год>`» (напр. «Июнь 2026»)
- Контрагент ← Контрагент
- Проект ← Проект
- Сумма с НДС 20% ← СуммаСНДС
- Сумма без НДС ← СуммаБезНДС
- Ставка НДС ← СтавкаНДС (**с преобразованием, см. ниже**)
- Комментарий ← Этап
- № документа ← НомерК7

### Лист «ДС» (в чате пользователь называет его «Оплата»; тип записи `Оплата`)
- Подразделение ← Подразделение
- Сценарий ← «ФАКТ»
- Период ← «`<Месяц> <Год>`»
- Контрагент ← Контрагент
- Проект ← Проект
- Сумма с НДС ← СуммаСНДС
- Сумма без НДС ← СуммаБезНДС
- Ставка НДС ← СтавкаНДС (с преобразованием)
- № документа ← НомерК7
- Комментарий ← Комментарий

### Лист «ФОТ» (тип записи `НачислениеЗарплаты`)
**Одна строка = один сотрудник.** Группировать записи по (Подразделение, Сотрудник):
- ФОТ ← сумма всех строк с ВидНачисленияЗП = «Оплата труда» (поле Оклад)
- Премия ← сумма всех строк с ВидНачисленияЗП = «Премия» (поле Премия)
- Комментарий ← объединить комментарии из строк с премией (через «; »)
- Строки ВидНачисленияЗП = «Административные расходы» — служебные, суммы не несут, игнорировать
- Подразделение можно писать как в выгрузке (в расчёте отчёта оно не используется, но для порядка оставить)

### Лист «Взаиморасчеты» (тип записи `Взаиморасчет`)
Это самый сложный лист. Порядок обработки:

**Шаг 1. Отсечь ВГО.** Исключить строки где ВГО = «Да» ИЛИ Направление содержит «ВГО» (значения «Доход ВГО», «Расход ВГО»). ВГО (внутригрупповые обороты) в отчёт по департаменту не входят.

**Шаг 2. Определить наименование отдела для отчёта.** Берётся из колонки **ПодразделениеКонтрагентДляОтчета** (это готовый маппинг из системы 1С). Возможные ситуации:
- Заполнено конкретным отделом («ОКП 3 (Кизилов), руб.», «Отдел продаж, руб.» и т.п.) → это **внутренний отдел офиса**.
- Содержит «др. офис» → это **другой офис**.
- **Пусто** → несоответствие, требует уточнения у пользователя (см. известные случаи ниже).

**Шаг 3. Обработать в зависимости от типа:**

- **Внутренний отдел офиса** (наименование «…, руб.»): взаиморасчёт в выгрузке продублирован в БДР и БДДС с одинаковой суммой. **Схлопнуть дубль — взять только одну строку** (по признаку ТипОборота = «БДР»; сумма БДР = БДДС, поэтому какую именно — неважно). В лист попадает одна строка на документ.

- **Другой офис** (Ташкент, Краснодар, Павелецкая, NFP и подобные): **делить по ТипОборота**:
  - ТипОборота = «БДР» → Отдел = «Взаиморасчеты с др. офисами по актам»
  - ТипОборота = «БДДС» → Отдел = «Взаиморасчеты с др. офисами по ДС»
  - **Не схлопывать**: документ с ВидВзаиморасчета = «Полноценный» имеет и БДР, и БДДС → идёт в обе строки (по актам И по ДС). Документ с ВидВзаиморасчета = «БДДС» имеет только БДДС → идёт только в «по ДС».
  - **В комментарий добавлять название офиса в начало**: `<ПодразделениеКонтрагент>. <исходный комментарий>` (напр. «Краснодар на Кожевенной. +ВЗР за Наталью…»). Это обязательно, т.к. в наименовании отдела офис не виден.

**Заполнение колонок листа:**
- Подразделение ← Подразделение
- Сценарий ← «ФАКТ»
- Период ← «`<Месяц> <Год>`»
- Отдел ← результат шага 2-3
- Контрагент ← Контрагент
- Проект ← Проект
- Направление ← **преобразование: Доход → «Исходящие», Расход → «Входящие»**
- Сумма без НДС ← СуммаБезНДС
- Комментарий ← см. правила выше (для др.офисов с названием офиса)

### Лист «Отчетность БИТ 2026» — фактический ФОТ (тип записи `ФактическийФОТ`)
Это единственная ячейка отчёта, заполняемая вручную (не формулой). В выгрузке за текущий месяц приходит фактический ФОТ за **предыдущий** месяц.
- Сумма = сумма поля «Сумма» по всем строкам `ФактическийФОТ`, **КРОМЕ** строки, где Сотрудник = «Сорокин Илья Вячеславович» И ВидНачисленияЗП = «Премия». (Оклад, отпуск, больничные и прочее по Сорокину — включаются; исключается только его премия.)
- Записать в ячейку столбца **предыдущего** месяца, строка «ФОТ фактический, руб.» (строка 46 отчёта). Пример: выгрузка за июнь содержит фактический ФОТ мая → писать в ячейку мая (столбец мая, строка 46). Для мая это ячейка `T46`.
- Фактический ФОТ текущего месяца остаётся пустым — придёт в следующей выгрузке.

## Преобразование ставки НДС (важно)
В листах Реализация и ДС ставку НДС из выгрузки писать так:
- «22%», «20%», «5%» → как есть
- «Без НДС» → **преобразовать в «0%»**

Причина: контрагенты с иностранным государством (экспорт, напр. ДарВетер) — это реализация услуг по ставке 0%, и она должна попадать в «Сумму реализации услуг». Для таких строк **Сумма с НДС = Сумма без НДС** (совпадают, НДС нулевой) — так и приходит в выгрузке, переносить обе одинаковыми.

**Не путать:** значения «Софт 1С» и «Софт БИТ» в ставке — это реализация ПО, идёт в отдельную строку «Реализация ПО». Их НЕ трогать и НЕ преобразовывать в 0%. На июнь таких не было, но если появятся — обрабатывать отдельно и при сомнении уточнить.

### Правка формул отчёта под 0%
В листе «Отчетность БИТ 2026» формулы столбца «С НДС» (реализация услуг, ПДС) суммируют только ставки 20%/5%/22%. Чтобы строки 0% попали в отчёт, в каждую такую формулу добавить слагаемое-копию с критерием «0%». Формул около 48 (по всем месяцам, строки реализации и ПДС). Механически: найти слагаемое `SUMIFS(…,"20%",…)`, продублировать его с заменой «20%»→«0%», добавить в конец формулы. Проверять, что «0%» ещё не добавлен (идемпотентность).

## Защита старых периодов (критично)
Старые периоды проверены финансовым департаментом и **не подлежат никаким изменениям**.

1. **Не перезаписывать ячейки старых периодов.** Новый месяц дописывать строго после последней существующей строки.

2. **Проблема «дата → число» (напр. 46143):** в старых периодах поле «Период» хранится как дата с форматом отображения (`d-mmm`, `m/d/yyyy`, `mmmm yyyy` — по-разному на разных листах). Нельзя допускать, чтобы дата превратилась в число. **Решение — вообще не трогать ячейки периодов старых месяцев.** Новый месяц писать **текстом** «`<Месяц> <Год>`» (напр. «Июнь 2026»), формат ячейки периода нового месяца ставить текстовый (`@`). Отчёт корректно матчит и текст, и дату (сравнение идёт по отображаемому значению).

3. **Хрупкие относительные формулы в телах листов.** В старых периодах встречаются формулы вроде `=I80*1.22`, `=I4+I4*J4`, `=460000-60000`. При удалении пустых строк они могут «разъехаться». Перед удалением пустых строк — **зафиксировать эти формулы их посчитанными значениями** (взять из пересчитанной копии, data_only). Это не меняет результат и защищает проверенные цифры. Затем удалять пустые строки и дописывать новый месяц.

4. **Сверка неизменности.** После заполнения сравнить исходные данные старых периодов (значения в листах-источниках) с оригиналом — они должны быть идентичны. Расхождений в старых месяцах быть не должно.

## Работа с таблицами Excel и фильтром

На листах Реализация/ДС/Взаиморасчеты/ФОТ есть «умные таблицы» Excel (Таблица1–Таблица4). Это источник двух проблем, которые нужно предотвращать:

1. **Ошибка при открытии (recoveryLog «Удалено свойство: Таблица/Автофильтр»):** возникает, если дописать строки за пределами диапазона таблицы или создать отдельный автофильтр, конфликтующий с таблицей. **Решение:** после заполнения обновить `ref` каждой таблицы точно под данные (заголовок + все строки), НЕ создавать отдельный `auto_filter` на листе, а фильтр задавать внутри таблицы (`autoFilter.ref` = тот же диапазон). Диапазон таблицы и её автофильтра должны совпадать.

2. **Фильтр по периоду.** На каждом из 4 листов установить фильтр по колонке «Период» = текущий месяц («`<Месяц> <Год>`»). Фильтр должен быть **применён физически** (строки других периодов реально скрыты при открытии, не просто «выбран в выпадающем списке»). Это делается через LibreOffice UNO (`rng.filter(...)` с полем периода). Проверять: после применения видимыми должны остаться только строки текущего месяца (число видимых = числу добавленных строк).
   - Важно: при применении фильтра через UNO поле в `TableFilterField.Field` отсчитывается относительно начала диапазона фильтра (если диапазон начинается со столбца B, то колонка «Период» E имеет индекс 3, а не 4).

## Порядок сборки файла (рекомендуемая последовательность)
1. Снять пароль с присланного Excel (LibreOffice UNO с параметром Password).
2. Распарсить MXL-выгрузку.
3. Пересчитать копию оригинала (для получения значений хрупких формул).
4. В рабочей книге: (а) поправить формулы отчёта под 0%; (б) зафиксировать хрупкие формулы тел листов значениями; (в) удалить пустые строки; (г) дописать новый месяц (период — текстом); (д) вписать фактический ФОТ предыдущего месяца в строку 46.
5. Обновить диапазоны таблиц, применить фильтр по периоду физически (UNO).
6. Пересчитать книгу, свериться с ошибками (должны совпадать с оригиналом — новых нет; `#DIV/0!` в столбцах «План» и ошибки на служебных листах «Отчетность»/«Калькулятор грейдов» — это фон оригинала, не наши).
7. Сохранить с паролем 987456, имя нового месяца.
8. **Показатели для сообщения считать самыми последними**, после всех операций (см. ниже).

## Итоговые показатели (выводить сообщением после заполнения)
После формирования файла вывести из отчёта БИТ:
- Сумма актов с учётом НДС и без учёта НДС
- Сумма оплат с учётом НДС и без учёта НДС
- Сумма маржи по актам
- Сумма маржи по ДС
- ФОТ (расчётный)
- Прибыль по ДС (строка «Сверка отдела, руб»)
- Прибыль по актам (строка «Прибыль отдела, мес, к зачету»)

**Расположение в отчёте (столбцы месяца, блок из 4 колонок на месяц: С НДС/Факт, Без НДС/Факт, С НДС/План, Без НДС/План):**
- Сумма актов = строка 8 «Сумма реализации»: с НДС из столбца «С НДС/Факт», без НДС из столбца «Без НДС/Факт»
- Сумма оплат = строка 7 «Поступление ДС»: аналогично
- Маржа по актам = строка 42 «Валовая прибыль без НДС (маржа отдела)»
- Маржа по ДС = строка 43 «Приход ДС маржа»
- ФОТ расчётный = строка 45
- Прибыль по ДС = строка 67 «Сверка отдела, руб»
- Прибыль по актам = строка 78 «Прибыль отдела, мес, к зачету»

**Важный нюанс расчёта прибыли.** Прибыль зависит от «Дельты ФОТ» = (ФОТ фактический − ФОТ расчётный) предыдущего месяца. ФОТ расчётный старых месяцев считается формулой SUMIFS по листу ФОТ с текстовым критерием «`<Месяц> <Год>`». В листе ФОТ старые периоды хранятся как **даты** с форматом `m/d/yyyy` (отображаются «5/1/2026», а не «Май 2026»). **Excel матчит дату с текстовым критерием, а LibreOffice — нет.** Поэтому при пересчёте в LibreOffice ФОТ расчётный старого месяца может выйти = 0, что искажает дельту и прибыль. 

Следствие: **прибыль, посчитанная в LibreOffice, ненадёжна** — в самом файле формулы правильные, и в Excel у пользователя посчитается верно. Для вывода прибыли в сообщении — досчитывать вручную по цепочке формул с корректным ФОТ расчётным предыдущего месяца (сумма ФОТ+Премия по строкам предыдущего месяца листа ФОТ, сценарий ФАКТ), а не брать значение из пересчёта LibreOffice.

Цепочка прибыли (для ручного контроля):
- ФОТ расчётный предыдущего месяца = сумма (ФОТ + Премия) по строкам того месяца, ФАКТ
- Дельта ФОТ = ФОТ фактический пред. мес. − ФОТ расчётный пред. мес.
- ФОТ с учётом дельты = ФОТ расчётный текущий + Дельта
- Себестоимость отдела = ФОТ с дельтой + налог на ФОТ (20% от ФОТ с дельтой) + административные затраты
- Прибыль по ДС = Маржа ДС − Себестоимость
- Прибыль по актам = Маржа акты − Себестоимость

## Обязательная самопроверка перед выдачей
1. Суммы актов и оплат (с НДС и без) сходятся с ручным подсчётом из выгрузки.
2. Маржа по актам и по ДС — при наличии контрольных цифр от пользователя сверить; расхождение чаще всего означает попавшее по ошибке ГКП 10.6 или не отсечённый не-ФАКТ/ВГО.
3. Взаиморасчёты: итог и разбивка по отделам; при наличии контрольного списка — сверить по документам.
4. Старые периоды не изменились; периоды не превратились в числа.
5. Фильтр по текущему месяцу применён и показывает только нужные строки.
6. Таблицы валидны (ref = autoFilter ref), файл откроется без recoveryLog.
7. Фактический ФОТ предыдущего месяца записан в правильную ячейку (без премии Сорокина).

## Известные соответствия и решённые случаи (справочно)
- ДКП N (ФИО) в выгрузке ↔ ОКП N (ФИО) в отчёте — маппинг берётся из колонки ПодразделениеКонтрагентДляОтчета, не выводить руками.
- ДСКК (Миронов) ↔ ОСКК (Миронов), руб. — тоже из ДляОтчета.
- БИТ.Цифра (в выгрузке ДляОтчета пусто) → относить в «Другие отделы (QV, АРТ, ОРБ), руб.» (внутренний отдел, схлопывать дубль).
- Офис Ташкент, Павелецкая (ДляОтчета пусто) → другие офисы (делить по актам/ДС). Павелецкая — др. офис, несмотря на то что БДР=БДДС.
- Краснодар на Кожевенной, NFP → другие офисы.

**При появлении НОВОГО отдела/офиса с пустым «ДляОтчета» — обязательно спросить пользователя, куда его относить.** Не додумывать по аналогии.
