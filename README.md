Zapolnyator — система автоматического заполнения отчёта ДКП
Назначение
Система автоматизирует ежемесячное заполнение отчёта ДКП (департамент корпоративных продаж) на основе выгрузки из 1С. Пользователь отправляет через Telegram два файла: (1) Excel-шаблон отчёта за предыдущий месяц и (2) файл с данными за новый месяц (XLSX или MXL). ИИ-агент анализирует данные, применяет бизнес-правила, дописывает новый месяц в отчёт, вычисляет показатели и возвращает заполненный Excel-файл.

Архитектура
Система состоит из трёх независимых микросервисов, взаимодействующих через REST API и Redis:

Сервис	Назначение	Технологии
Telegram Gateway	Приём файлов, управление диалогом, отправка уведомлений	aiogram, aiohttp, Redis
Orchestrator	Управление бизнес-процессом, вызов DeepSeek (Function Calling), управление сессиями, долгосрочная память (SQLite), координация Worker	FastAPI, DeepSeek SDK, SQLite, Redis
Worker Engine	Выполнение тяжёлых вычислений: чтение данных (XLSX/MXL), фильтрация, запись Excel, расчёт метрик	FastAPI, openpyxl, pandas, msoffcrypto-tool
Схема взаимодействия
Пользователь отправляет файлы → Telegram Gateway → создаёт задачу в Redis и уведомляет Orchestrator через REST.

Orchestrator забирает задачу, управляет диалогом с DeepSeek, вызывает инструменты Worker по REST.

При необходимости задать вопрос – Orchestrator отправляет сообщение через Gateway, получает ответ и продолжает.

По завершении Orchestrator передаёт путь к готовому файлу и показатели в Gateway, который отправляет их пользователю.

Сервис Telegram Gateway
Назначение
Принимает команды /start, /help, /reset, /approve, /cancel, /stop.

Принимает файлы: Excel-шаблон (.xlsx/.xls) и файл с данными (.xlsx/.xls/.mxl) в любом порядке.

Сохраняет файлы на сервер во временную папку.

Отправляет пользователю уведомления о ходе обработки, ошибки, готовый файл и показатели.

Опрашивает Orchestrator о статусе задачи и отображает прогресс.

Основные файлы и методы
main.py
main() – точка входа: создаёт экземпляр бота, диспетчер, регистрирует обработчики и запускает polling.

handlers.py
cmd_start() – обработчик /start, сбрасывает состояние и выводит инструкцию.

cmd_reset() – обработчик /reset, очищает состояние пользователя.

cmd_approve() – одобряет запрос к LLM (продолжает обработку).

cmd_cancel() – отменяет запрос к LLM.

cmd_stop() – принудительно останавливает текущую задачу.

handle_document() – обрабатывает загруженные файлы: определяет тип (Excel-шаблон, Excel-данные или MXL-данные), сохраняет и ожидает второй файл.

start_processing() – вызывает Orchestrator для создания задачи, сохраняет task_id и запускает фоновый опрос статуса.

poll_task_status() – периодически опрашивает статус задачи, реагирует на изменения (done, error, waiting_question) и отправляет сообщения пользователю.

handle_answer() – обрабатывает ответ пользователя на уточняющий вопрос.

save_document() – скачивает файл из Telegram и сохраняет в temp папку.

register_handlers() – регистрирует роутер в диспетчере.

client.py
OrchestratorClient – клиент для взаимодействия с Orchestrator по REST.

create_task() – отправляет POST /api/v1/task с путями к файлам, месяцем и годом.

get_task_status() – GET /api/v1/task/{task_id} – получает статус задачи.

answer_question() – POST /api/v1/task/{task_id}/answer – отправляет ответ пользователя.

approve_llm(), cancel_llm(), stop_task() – методы для управления выполнением LLM-запросов.

Сервис Orchestrator
Назначение
Координирует выполнение бизнес-процесса.

Управляет сессиями пользователей через Redis (состояние, история диалога, метаданные).

Хранит долгосрочную память (правила маппинга, исключения) в SQLite.

Вызывает DeepSeek API с системным промптом (полная инструкция) и доступными инструментами.

Обрабатывает запросы на вызов инструментов (фильтрация данных, запись Excel, чтение структуры).

При неопределённости задаёт уточняющие вопросы пользователю.

Перед каждым запросом в LLM отправляет пользователю файл с промптом и ожидает одобрения (/approve или /cancel).

Логирует использование токенов и отправляет диагностическую информацию в Telegram.

Основные файлы и методы
main.py
FastAPI приложение, подключает роутер API.

api.py
create_task() – POST /api/v1/task – инициирует новую задачу, сохраняет начальное состояние в Redis, запускает агента в фоне.

get_task_status() – GET /api/v1/task/{task_id} – возвращает текущий статус, результат или ошибку.

answer_question() – POST /api/v1/task/{task_id}/answer – передаёт ответ пользователя агенту.

approve_llm() – POST /api/v1/task/{task_id}/approve – одобряет запрос к LLM.

cancel_llm() – POST /api/v1/task/{task_id}/cancel – отменяет запрос.

stop_task() – POST /api/v1/task/{task_id}/stop – принудительно останавливает задачу.

agent.py
OrchestratorAgent – основной класс, управляющий циклом ИИ-агента.

__init__() – загружает инструкцию из INSTRUCTION.md, инициализирует DeepSeek-клиент.

run_agent_cycle() – основной метод: определяет тип данных (XLSX или MXL), читает структуру Excel, формирует системный промпт, сохраняет промпт в файл и отправляет пользователю для утверждения.

approve_llm_request() – после /approve запускает цикл взаимодействия с LLM.

cancel_llm_request() – отменяет запрос.

_run_llm_cycle() – цикл вызовов DeepSeek: отправляет сообщения, обрабатывает вызовы инструментов, обрабатывает ответы, задаёт уточняющие вопросы, завершает задачу.

process_answer() – обработчик ответа пользователя на уточняющий вопрос (возобновляет цикл).

stop_task() – принудительная остановка.

_send_telegram_message() – отправляет сообщение и/или файл пользователю через Telegram API.

_save_debug_data() – сохраняет данные во временный файл.

_sanitize_for_json() – очищает строки от управляющих символов.

session_manager.py
Управление сессиями в Redis.

init_session() – создаёт запись для новой задачи.

get_session() – возвращает состояние задачи.

update_session() – обновляет состояние.

delete_session() – удаляет сессию.

memory.py
Долгосрочная память (SQLite).

save_rule() – сохраняет правило маппинга колонок или исключения.

get_rule() – получает правило по ключу.

load_all_rules() – загружает все правила для передачи в Worker.

worker_client.py
Клиент для вызова инструментов Worker.

call_tool(tool, arguments) – отправляет POST /api/v1/tool с именем инструмента и аргументами.

Сервис Worker Engine
Назначение
Предоставляет REST API для выполнения ресурсоёмких операций:

Чтение данных из XLSX или MXL.

Фильтрация данных по критериям.

Чтение структуры Excel-шаблона.

Запись данных в Excel с обновлением таблиц, автофильтром и защитой.

Расчёт итоговых показателей (пока заглушка, будет реализовано).

Не содержит бизнес-логики — только примитивные операции.

Основные файлы и методы
main.py
FastAPI приложение, эндпоинт /api/v1/tool.

call_tool() – диспетчер инструментов: по имени вызывает соответствующую функцию и возвращает унифицированный ответ.

mxl_parser.py
parse_mxl() – парсит MXL-файл (текстовый формат 1С), определяет кодировку, извлекает заголовки и данные.

convert_mxl_to_csv() – конвертирует MXL в CSV с кодировкой UTF-8 (с BOM для корректного отображения в Telegram), сохраняет рядом с исходным файлом и возвращает путь и метаданные.

excel_processor.py
read_excel_structure() – читает структуру Excel-файла (листы, заголовки, количество строк) для анализа агентом.

write_excel_data() – записывает данные на указанные листы, обновляет таблицы, устанавливает автофильтр, сохраняет с паролем (опционально).

Инструменты (вызываются через /api/v1/tool):
Имя инструмента	Описание	Аргументы	Возврат
read_excel_data	Читает первый лист XLSX и возвращает данные в виде списка словарей	file_path (обязательно), sheet_name (опционально)	data, columns, rows
convert_mxl_to_csv	Конвертирует MXL в CSV, сохраняет файл и возвращает данные в виде строки	file_path	csv_data, csv_path, columns, rows, size_bytes
filter_mxl_data	Фильтрует данные (CSV или MXL) по критериям	file_path, filters (словарь колонка→значение)	filtered_data, count
read_excel_structure	Читает структуру Excel-шаблона	file_path	sheets (листы с заголовками)
write_excel	Записывает данные в Excel и сохраняет файл	template_path, sheets_data, password (опционально), output_path (опционально)	output_path
Установка и развёртывание
Требования
Ubuntu 24.04 (или Docker-контейнеры)

Python 3.11+

Docker и Docker Compose

Telegram Bot Token (получить у @BotFather)

DeepSeek API Key (получить на platform.deepseek.com)

(Опционально) OpenAI API Key

Локальная разработка
Клонируйте репозиторий:

bash
git clone git@github.com:greenscale93/Zapolnyator.git
cd Zapolnyator
Создайте файл .env на основе .env.example (пример есть в репозитории):

text
TELEGRAM_BOT_TOKEN=ваш_токен
DEEPSEEK_API_KEY=ваш_ключ
TEMP_DIR=./temp
ORCHESTRATOR_URL=http://orchestrator:8000
WORKER_URL=http://worker:8000
REDIS_URL=redis://redis:6379
TELEGRAM_CHAT_ID=ваш_чат_ид  # для уведомлений о деплое (опционально)
Убедитесь, что у вас установлены Docker и Docker Compose.

Запустите:

bash
docker compose up -d --build
Проверьте логи:

bash
docker compose logs -f
Автоматический деплой на VPS (GitHub Actions)
Настроен workflow .github/workflows/deploy.yml при пуше в main.

В GitHub Secrets должны быть заданы:

VPS_HOST – IP сервера

VPS_USER – имя пользователя (обычно root)

VPS_SSH_KEY – приватный SSH-ключ

TELEGRAM_BOT_TOKEN

DEEPSEEK_API_KEY

TELEGRAM_CHAT_ID – для уведомлений о статусе деплоя

DOCKER_USERNAME и DOCKER_PASSWORD – для авторизации в Docker Hub (чтобы избежать лимитов)

Ручное обновление на VPS
Если деплой упал, можно обновить вручную:

bash
ssh root@ваш_IP
cd /home/root/Zapolnyator
git pull origin main
docker compose down
docker compose up -d --build
Использование
Команды бота
/start – начало работы, сброс состояния.

/reset – сброс состояния.

/approve – одобрить запрос к LLM (после получения файла с промптом).

/cancel – отменить запрос к LLM.

/stop – принудительно остановить текущую задачу.

Порядок загрузки файлов
Отправьте Excel-шаблон (отчёт за предыдущий месяц) – бот подтвердит.

Отправьте файл с данными (XLSX или MXL) – бот подтвердит.

Можно отправлять в любом порядке.

После загрузки обоих файлов бот создаст задачу и начнёт обработку.

Диагностика
Каждый шаг обработки сопровождается уведомлением в Telegram.

Перед отправкой запроса в DeepSeek бот присылает файл с системным промптом – вы можете изучить его и решить, отправлять или отменить.

При ошибках или больших объёмах данных бот присылает диагностические файлы (история, результаты фильтрации).

Переход на XLSX
Рекомендуется использовать XLSX вместо MXL. Это ускоряет обработку, исключает проблемы с кодировкой и упрощает отладку. Бот поддерживает оба формата, но приоритет – XLSX.

Логирование и мониторинг
Все сервисы пишут логи в stdout (видны через docker compose logs).

Общий correlation_id (task_id) используется для трассировки запросов через все сервисы.

В Telegram отправляются сообщения о ключевых событиях: создание задачи, ошибки, завершение, большие файлы.
